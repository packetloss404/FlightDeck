FeaturesTwo products, one binary.

FlightDeck is a coding agent first. The orchestration surface that originally shipped with it — Flights composed of Milestones and Tasks — is still there, one keystroke away, for when a single chat doesn't fit the shape of the work.

Native agent

Streaming chat against your cwd

Anthropic Messages API with SSE streaming, extended thinking support, and clean token-delta rendering. Text shows up as it's generated; tool calls appear inline as the model decides to make them.

Tools

Six in-process primitives

Sandboxed to the project directory. No shell-escaping mistakes, no accidental `..`.

  • read · line-numbered file reads
  • write · create / overwrite
  • edit · exact string replacement
  • bash · gated shell execution
  • grep · regex over the tree
  • glob · pattern file match
Safety

Approval gates & doom-loop halt

bash and writes outside the project dir trigger a centered banner — y/a approve, n/d/Esc deny. If the model calls the same tool three times with identical input, the runner halts and feeds the denial back as a tool result so it can course-correct.

Providers

OS-keyring secrets, in-TUI setup

Add a provider from inside the TUI: paste your API key, pick a model, test against /v1/models, save. The key lands in the OS keyring (Windows Credential Manager, macOS Keychain, or libsecret on Linux). Nothing sensitive ever touches providers.json; in-memory copies are wrapped in Zeroizing<String>.

Context

Compaction without losing the tail

Token-budget pruner at 180k total / 40k protected tail. Older turns collapse to a summary placeholder while the most recent tool results stay intact — so the model keeps full fidelity where it matters.

Orchestration

Flight Deck overlay (Ctrl+F)

When a single chat isn't the right shape, drop into the overlay: Flights composed of Milestones composed of Tasks, each assigned to the native agent or an external CLI (Claude Code, Codex, Gemini, OpenCode). Parallel execution, milestone gating, review-before-continue.

TUI

Ratatui, keyboard-driven

Fuzzy command palette (Ctrl+P), context-sensitive help (?), five built-in themes, custom themes from ~/.flightdeck/themes/, tabbed sessions, diff overlay, toast notifications.

Runtime

Sync TUI, async agent thread

The render loop stays synchronous; each native task spawns a dedicated OS thread with a current-thread Tokio runtime. Events flow back over std mpsc, the Agent view and Sessions view pull from the same channel.

Ship

One binary, zero deps

No Node. No Tauri. No sidecar. cargo build --release produces a ~9 MB executable that runs against ConPTY on Windows or POSIX PTY elsewhere.

QuickstartThree commands and a key.

Requires Rust 1.70+ and a working OS keyring. First launch opens into the Agent view; configure an Anthropic provider and you're streaming.

  1. Clone the repo and cargo build --release.
  2. Run the binary; it opens to the Agent view.
  3. Press Ctrl+P, type provider add, paste your Anthropic key.
  4. Back in the Agent view, type your first request and press Enter.

State lives in ~/.packetcode/ (legacy path from the fork — will migrate to ~/.flightdeck/ in a later release). API keys live in your OS keyring under service=flightdeck, account=provider:<id>.

# clone + build git clone git@github.com:packetloss404/flightdeck.git cd flightdeck cargo build --release # run (Unix) ./target/release/flightdeck # run (Windows) .\target\release\flightdeck.exe # tests (73, deterministic, no network) cargo test

Your keys stay in the keyring.

FlightDeck never writes API keys to disk in its own state files. The provider registry holds metadata only; secrets go through the OS-native keyring and are zeroized in memory when the form closes or the provider client drops.

ArchitectureOne dispatch fork, one event channel.

Two agent kinds coexist cleanly. The orchestrator's tick() returns a TaskDispatch enum — Pty for external CLIs, Native for the in-process agent. Both emit PtyEvents into the same channel the TUI already drains, so the Sessions view renders both kinds identically.

Design choicesAdditive, not revolutionary.

The native-agent stack is a new module. Everything downstream — the Flight data model, the Sessions view, the PTY output parsing, the persistence layer — stays exactly the same. Adding a new agent backend is one enum variant and one match arm.

Invariant 1

One dispatch point

Orchestrator::tick() is the single place where Flights become work. New backends slot in as enum variants on TaskDispatch.

Invariant 2

Event channel reuse

The native runner emits PtyEvent::Output and PtyEvent::Exit over the same mpsc the TUI already drains. The Sessions view doesn't know where the bytes came from.

Invariant 3

Keyring-only secrets

No api_key field anywhere in the persisted model. Keys live in the OS keyring; in-memory copies are Zeroizing<String> and Drop-zeroed on form close.

Invariant 4

Approval as a channel

When a tool needs approval, the chat driver sends an ApprovalQuery with a oneshot::Sender<bool>, then awaits up to 300s. The app drains the channel, shows the banner, intercepts y/n, and sends the decision back.

RoadmapWhat's next.

v0.2 is the first cut with the new native-agent stack. The follow-ups below are intentionally deferred — the architecture is already set up for them.

v0.3

More providers

OpenAI and Google behind the existing Provider trait. Optional local models via Ollama.

v0.3

More tools

webfetch, multiedit, apply_patch, plus an LSP bridge for type-aware edits.

v0.3

MCP client

Talk to any Model Context Protocol server; registered tools flow into the existing ToolRegistry.

Polish

Data-dir rename

Move state from ~/.packetcode/ to ~/.flightdeck/ with a one-time migration.

Polish

Real cancellation

A cancellation channel into the chat thread so Ctrl+C during a turn actually kills the provider stream.

Polish

Runner provider injection

Refactor the Flight-mode runner to take a Provider trait object, matching chat-mode — so Flight tests can use MockProvider too.
